# Orchestration Log — 2026-04-13T10:56Z

## Agent: Frodo

**Task:** Fix language switch reactivity bug  
**Status:** SUCCESS  

### Summary
Frodo identified and resolved a critical bug where switching language in Settings didn't reactively update already-open views (saint details, category saint lists). Users had to navigate away and back to see updated content.

### Root Cause
- `SaintDetailView` received `let saint: Saint` (a captured value-type snapshot) at navigation time
- `CategorySaintsListView` similarly received `let saints: [Saint]`  
- When `SaintListViewModel.loadData()` reloaded saints for the new language, detail views held stale structs
- Value types don't automatically update when the source data changes

### Solution
- Changed `SaintDetailView` to accept `saintId: String + viewModel: SaintListViewModel` and compute saint reactively
- Changed `CategorySaintsListView` to accept `groupId`/`valueId + viewModel` and compute saints reactively
- Updated all navigation sites (`SaintListView`, `SearchView`, `CategoryBrowseView`) to pass saint IDs and viewModel references
- Changed navigation destinations from `.navigationDestination(for: Saint.self)` to `.navigationDestination(for: String.self)`

### Pattern Established
**Never pass captured content model values to detail views when that content can change.** Always pass an ID + @Observable data source for reactive re-rendering.

### Verification
- ✅ Build verified clean on iOS 17 simulator
- ✅ Language switching now updates detail views reactively without re-navigation
- ✅ No conflicts with existing navigation patterns
